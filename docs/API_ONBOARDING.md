## What This API Does

The ModelSEED API is a Python FastAPI backend for metabolic modeling. It replaces the legacy Perl-based ProbModelSEED JSON-RPC service with a modern REST API.

**Core capabilities:**

- **Model CRUD** -- List, view, copy, delete, and export metabolic models stored in the PATRIC workspace
- **Model Editing** -- Add, remove, and modify reactions, compounds, and biomass objectives in a model (atomic operations with full validation)
- **Model Reconstruction** -- Build a metabolic model from a BV-BRC genome ID (fetches genome, annotates, builds draft model)
- **Gapfilling** -- Fill gaps in a model so it can produce biomass on a given media condition
- **Flux Balance Analysis (FBA)** -- Simulate growth by optimizing flux through the metabolic network
- **Gapfill Management** -- Integrate, unintegrate, or delete gapfill solutions from a model
- **Biochemistry Queries** -- Search and look up compounds and reactions from the ModelSEED biochemistry database (public, no auth)
- **Media Browsing** -- List public and user-defined growth media formulations
- **Workspace Proxy** -- Full proxy to the PATRIC workspace service (ls, get, create, copy, delete, metadata, permissions)

**What it does NOT do:**

- Does not run long-running computations in-process. Reconstruction, gapfilling, and FBA are dispatched as async jobs (subprocess or Celery) and polled via `GET /api/jobs`.
- Does not serve the production frontend. A separate Next.js/TypeScript app (built by the frontend team) will consume this API. A lightweight demo dashboard at `/demo/` exists for development testing only.

## Authentication

Most endpoints require a **PATRIC authentication token** in the `Authorization` header.

### Getting a Token

1. Log in to https://www.bv-brc.org
2. Open browser console (F12)
3. Run: `copy(TOKEN)`
4. Use the copied string as the token

### Sending the Token

```
Authorization: un=username|tokenid=...|sig=...
```

The API accepts the token in the `Authorization` or `Authentication` header. It automatically strips `Bearer ` prefix and surrounding quotes if present.

### Which Endpoints Need Auth?

| Group | Auth Required |
|-------|--------------|
| `/api/health` | No |
| `/api/biochem/*` | **No** (public) |
| `/api/models/*` | Yes |
| `/api/jobs/*` | Yes |
| `/api/media/*` | Yes |
| `/api/workspace/*` | Yes |

Unauthenticated requests to protected endpoints return **HTTP 401**.

## Domain Concepts

Understanding these concepts is essential for working with the API.

### Workspace References

Models, media, and other objects are stored in the **PATRIC workspace** (a cloud filesystem). Objects are referenced by path:

```
/{username}/modelseed/MyModel          # a model folder
/{username}/modelseed/MyModel/model    # the model data sub-object
/chenry/public/modelsupport/media      # public media folder
```

Most API parameters named `ref` or `model` expect a workspace path like `/{username}/modelseed/ModelName`.

### Metabolic Models

A metabolic model is a mathematical representation of an organism's metabolism:

- **Reactions** -- Chemical transformations (e.g., glucose + ATP -> glucose-6-phosphate + ADP). Each has a direction (`>` forward only, `<` reverse only, `=` reversible) and optional gene-protein-reaction (GPR) association.
- **Compounds (Metabolites)** -- Chemical species that participate in reactions. Located in specific compartments.
- **Compartments** -- Cellular locations: `c0` (cytoplasm), `e0` (extracellular), `p0` (periplasm), etc.
- **Biomass** -- A pseudo-reaction representing growth. Lists all compounds the cell needs to produce to grow, with stoichiometric coefficients.
- **Genes** -- Derived from GPR associations on reactions. Not stored directly; computed from the reaction data.

### ID Conventions

| Entity | Biochem DB ID | Model ID (with compartment) |
|--------|--------------|----------------------------|
| Reaction | `rxn00001` | `rxn00001_c0` |
| Compound | `cpd00001` | `cpd00001_c0` |
| Biomass | -- | `bio1` |
| Compartment | `c` | `c0` |

The biochemistry database uses plain IDs (`rxn00001`). Model objects append a compartment index (`_c0`). When adding reactions/compounds via the edit API, you provide the biochem ID and compartment separately.

### Gapfilling

Draft models often cannot produce biomass because they are missing reactions. **Gapfilling** finds a minimal set of reactions to add from the biochemistry database so the model can grow on a specified media. Gapfill solutions can be:

- **Integrated** -- The added reactions are active in the model
- **Unintegrated** -- The solution exists but reactions are not active
- **Deleted** -- The solution is removed entirely

### Media

A **media condition** defines what nutrients are available to the organism (which compounds can be imported from the environment). Media are stored as workspace objects. Common examples: Complete media, Minimal media, Carbon-source-specific media.

### Jobs

Long-running operations (reconstruct, gapfill, FBA) run as **async jobs**. The flow:

1. POST to `/api/jobs/{operation}` -- returns a `job_id` string
2. Poll `GET /api/jobs?ids={job_id}` -- returns status (`queued`, `in-progress`, `completed`, `failed`)
3. When `completed`, the result is available (model saved to workspace, FBA results in job record)

#### `POST /api/models/gapfills/manage` -- Manage Gapfills

**Request Body:**

```json
{
  "model": "/user/modelseed/MyModel",
  "commands": {
    "gf.0": "I",
    "gf.1": "U",
    "gf.2": "D"
  },
  "selected_solutions": {
    "gf.0": 0
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `model` | string | Model workspace ref |
| `commands` | object | Map of gapfill_id to action: `"I"` (integrate), `"U"` (unintegrate), `"D"` (delete) |
| `selected_solutions` | object | Map of gapfill_id to solution index (for integrate; defaults to 0) |

**Behavior:**
- **Integrate (`I`)**: Adds gapfill reactions to the model. If a reaction already exists with a different direction, widens it to reversible (`=`).
- **Unintegrate (`U`)**: Removes gapfill reactions from the model. Narrows reaction directions back if possible, or removes entirely if the reaction was only added by this gapfill.
- **Delete (`D`)**: Unintegrates first if needed, then removes the gapfill record and associated workspace objects.

All operations are atomic -- if any gapfill ID is not found, the entire request fails with HTTP 400.

### Model Editing

#### `POST /api/models/edit` -- Edit Model

The most powerful endpoint. Performs **atomic** edits: validates everything first, then mutates the model, then saves once. If any validation fails, nothing is changed.

**Request Body:**

```json
{
  "model": "/user/modelseed/MyModel",
  "reactions_to_add": [
    {
      "reaction_id": "rxn00001",
      "compartment": "c0",
      "direction": ">",
      "gpr": "gene1 or (gene2 and gene3)"
    }
  ],
  "reactions_to_remove": ["rxn00100_c0"],
  "reactions_to_modify": [
    {
      "reaction_id": "rxn00050_c0",
      "direction": "=",
      "gpr": "geneA"
    }
  ],
  "compounds_to_add": [
    {
      "compound_id": "cpd00500",
      "compartment": "c0",
      "name": "Custom Compound"
    }
  ],
  "compounds_to_remove": ["cpd00999_c0"],
  "compounds_to_modify": [
    {
      "compound_id": "cpd00001_c0",
      "charge": -1
    }
  ],
  "biomass_changes": [
    {
      "biomass_id": "bio1",
      "compound_changes": [
        {"compound_id": "cpd00001_c0", "coefficient": -50.0},
        {"compound_id": "cpd00999_c0", "coefficient": 0}
      ]
    }
  ],
  "biomasses_to_add": [
    {
      "name": "Biomass 2",
      "compounds": [
        {"compound_id": "cpd00001_c0", "coefficient": -10.0}
      ]
    }
  ],
  "biomasses_to_remove": ["bio2"]
}
```

All arrays default to `[]` -- you can include only the operations you need.

**Field Reference:**

| Field | Type | Description |
|-------|------|-------------|
| `model` | string | Workspace ref (required) |
| `reactions_to_add` | array | Add reactions from biochem DB |
| `reactions_to_remove` | array of strings | Model reaction IDs to remove |
| `reactions_to_modify` | array | Modify existing model reactions |
| `compounds_to_add` | array | Add compounds from biochem DB |
| `compounds_to_remove` | array of strings | Model compound IDs to remove |
| `compounds_to_modify` | array | Modify existing model compounds |
| `biomass_changes` | array | Modify compounds in existing biomasses |
| `biomasses_to_add` | array | Add new biomass objectives |
| `biomasses_to_remove` | array of strings | Biomass IDs to remove |

**ReactionToAdd fields:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `reaction_id` | string | required | Biochem DB ID (e.g., `"rxn00001"`) |
| `compartment` | string | `"c0"` | Target compartment |
| `direction` | string | null | `">"`, `"<"`, `"="` -- null uses DB default |
| `gpr` | string | null | Gene-protein-reaction rule (e.g., `"gene1 or (gene2 and gene3)"`) |

**ReactionToModify fields:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `reaction_id` | string | required | Model reaction ID (e.g., `"rxn00001_c0"`) |
| `direction` | string | null | New direction |
| `name` | string | null | New display name |
| `gpr` | string | null | New GPR (empty string `""` clears it) |

**CompoundToAdd fields:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `compound_id` | string | required | Biochem DB ID (e.g., `"cpd00001"`) |
| `compartment` | string | `"c0"` | Target compartment |
| `name` | string | null | Override DB name |
| `formula` | string | null | Chemical formula |
| `charge` | float | null | Charge |

**CompoundToModify fields:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `compound_id` | string | required | Model compound ID (e.g., `"cpd00001_c0"`) |
| `name` | string | null | New name |
| `formula` | string | null | New formula |
| `charge` | float | null | New charge |

**BiomassChange fields:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `biomass_id` | string | required | Biomass ID (e.g., `"bio1"`) |
| `name` | string | null | New biomass name |
| `compound_changes` | array | `[]` | List of `{compound_id, coefficient}` changes. Coefficient `0` removes the compound from biomass. |

**BiomassToAdd fields:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | string | `"New Biomass"` | Biomass name |
| `compounds` | array | `[]` | Initial compounds as `{compound_id, coefficient}` |

**Response:**

```json
{
  "model": "/user/modelseed/MyModel",
  "reactions_added": ["rxn00001_c0"],
  "reactions_removed": ["rxn00100_c0"],
  "reactions_modified": ["rxn00050_c0"],
  "compounds_added": ["cpd00500_c0", "cpd00012_c0"],
  "compounds_removed": ["cpd00999_c0"],
  "compounds_modified": ["cpd00001_c0"],
  "biomasses_added": ["bio2"],
  "biomasses_modified": ["bio1"],
  "biomasses_removed": [],
  "warnings": ["Some warning message"]
}
```

**Key behaviors:**

1. **Auto-compound creation**: When adding a reaction, any compounds in the reaction's stoichiometry that don't exist in the model are automatically created from the biochem DB. These show up in `compounds_added`.
2. **Auto-compartment creation**: If the target compartment doesn't exist, it's created with default pH 7.0 and a standard name (c=Cytoplasm, e=Extracellular, p=Periplasm, etc.).
3. **Compound removal validation**: You cannot remove a compound that is still used by any reaction. The API returns HTTP 400 listing which reactions use it.
4. **GPR parsing**: GPR strings like `"gene1 or (gene2 and gene3)"` are parsed into the workspace's nested protein/subunit/feature structure automatically.
5. **Biomass coefficient 0 = remove**: Setting a compound's coefficient to `0` in `biomass_changes` removes it from that biomass.

**Validation errors** return HTTP 400 with all errors concatenated:

```json
{"detail": "Reaction rxn00001_c0 already exists; Compound cpd99999 not found in biochemistry database"}
```

#### `POST /api/jobs/reconstruct` -- Build Model from Genome

Builds a draft metabolic model from a BV-BRC genome. Optionally gapfills it in the same job.

**Request Body:**

```json
{
  "genome": "83332.12",
  "template_type": "gn",
  "atp_safe": true,
  "gapfill": true,
  "media": "/chenry/public/modelsupport/media/Complete",
  "output_path": "/user/modelseed/MyNewModel"
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `genome` | string | required | BV-BRC genome ID (e.g., `"83332.12"`) |
| `template_type` | string | `"gn"` | Template: `"gn"` (gram-negative), `"gp"` (gram-positive), `"core"` |
| `atp_safe` | bool | true | Apply ATP safety check |
| `gapfill` | bool | false | Gapfill after reconstruction |
| `media` | string | null | Workspace ref for gapfilling media |
| `output_path` | string | null | Where to save the model in workspace |

**Response:** `string` -- the job ID. Poll `GET /api/jobs?ids={job_id}` for status.

**What the job does:**
1. Fetches genome data from BV-BRC API (sequences, features, annotations)
2. Converts to KBase genome format with ontology events
3. Loads core + gram-specific templates from local JSON files
4. Builds draft model via `MSReconstructionUtils.build_metabolic_model()`
5. If `gapfill=true`, runs `MSGapfill.run_gapfilling()` and integrates the solution
6. Saves to workspace as a `modelfolder` with full metadata (organism name, taxonomy, domain, reaction/compound counts)

## Common Workflows

### Build and View a Model

```
1. POST /api/jobs/reconstruct
   Body: {"genome": "83332.12", "gapfill": true, "output_path": "/user/modelseed/Mtb"}
   -> Returns job_id

2. GET /api/jobs?ids={job_id}       (poll every 3-5 seconds)
   -> Wait for status == "completed"

3. GET /api/models/data?ref=/user/modelseed/Mtb
   -> Full model with reactions, compounds, genes, biomasses
```

### Edit a Model

```
1. POST /api/models/edit
   Body: {
     "model": "/user/modelseed/Mtb",
     "reactions_to_add": [{"reaction_id": "rxn00001", "compartment": "c0", "direction": ">"}],
     "reactions_to_remove": ["rxn00100_c0"]
   }
   -> Returns EditModelResponse with lists of what changed
```

### Gapfill and Manage Solutions

```
1. POST /api/jobs/gapfill
   Body: {"model": "/user/modelseed/Mtb", "media": "/chenry/public/modelsupport/media/Complete"}
   -> Returns job_id

2. GET /api/jobs?ids={job_id}       (poll until completed)

3. GET /api/models/gapfills?ref=/user/modelseed/Mtb
   -> List of solutions with integrated status

4. POST /api/models/gapfills/manage
   Body: {"model": "/user/modelseed/Mtb", "commands": {"gf.0": "I"}}
   -> Integrates solution gf.0
```

### Run FBA

```
1. POST /api/jobs/fba
   Body: {"model": "/user/modelseed/Mtb"}
   -> Returns job_id

2. GET /api/jobs?ids={job_id}       (poll until completed)

3. GET /api/models/fba?ref=/user/modelseed/Mtb
   -> List of FBA studies with objective values
```

## Architecture

```
Frontend (Next.js or demo dashboard)
    |
    v
FastAPI REST API (port 8000)
    |
    +-- /api/models/*       --> ModelService --> PATRIC Workspace (p3.theseed.org)
    +-- /api/models/edit    --> ModelService + BiochemService (local DB) --> Workspace
    +-- /api/workspace/*    --> WorkspaceService --> PATRIC Workspace (JSON-RPC proxy)
    +-- /api/biochem/*      --> BiochemService --> Local ModelSEEDDatabase JSON files
    +-- /api/media/*        --> WorkspaceService --> PATRIC Workspace
    +-- /api/jobs/*         --> JobDispatcher --> Subprocess or Celery+Redis
    |
Job Execution (async)
    |
    +-- reconstruct   --> BVBRCUtils + MSReconstructionUtils + MSGapfill
    +-- gapfill       --> FBAModelBuilder + MSGapfill
    +-- fba           --> cobra.Model.optimize()
    +-- merge         --> merge_models.py script
```

**Key design principles:**

1. **Thin layer** -- The API is a thin REST wrapper. Model operations work directly on workspace JSON dicts (no intermediate cobra/FBAModel conversion except for export and FBA). Biochem queries read pre-loaded JSON. Workspace calls are proxied directly.

2. **Async jobs** -- All computationally expensive operations run outside the API process. Two backends: subprocess (development, default) and Celery+Redis (production).

3. **Workspace as source of truth** -- Models, media, genomes, and gapfill solutions all live in the PATRIC workspace. The API reads from and writes to workspace; it has no local database.

4. **Atomic edits** -- Model editing validates all operations before mutating anything, then saves once. No partial writes.

## Error Handling

### HTTP Status Codes

| Code | Meaning | When |
|------|---------|------|
| 200 | Success | Normal response |
| 400 | Bad Request | Validation error (edit operations, invalid parameters) |
| 401 | Unauthorized | Missing or invalid PATRIC token |
| 403 | Forbidden | Workspace permission denied |
| 404 | Not Found | Workspace object doesn't exist |
| 500 | Server Error | Unhandled exception |
| 502 | Bad Gateway | Workspace service error (not permission/not-found) |

### Error Response Format

```json
{"detail": "Error message describing what went wrong"}
```

For edit operations, validation errors are concatenated with semicolons:

```json
{"detail": "Reaction rxn00001_c0 already exists; Unknown reaction ID rxn99999"}
```

### CORS

The API allows all origins by default (`["*"]`). Credentials, all methods, and all headers are permitted. This can be restricted via the `MODELSEED_CORS_ORIGINS` environment variable.
